# 功能更新与设计说明

本文档记录了关于“自动提取术语”功能的设计逻辑，以及相关界面的修复说明。

## 1. 界面交互修复：“幽灵”按钮变暗问题

**现象**：
在之前的版本中，当用户的鼠标悬浮或点击“全局术语表”输入框及左侧区域时，“自动提取术语”按钮会出现异常的变暗/Hover交互动画。

**根本原因**：
为了排版对齐，我们将“提取术语”的 `button` 和输入框 `textarea` 共同放置在了一个 HTML 的 `<label>` 标签内。根据 Web 标准行为，如果用户点击或划过 `<label>` 的任何区域（包括其内部的文字和输入框），浏览器会自动将焦点或交互事件传递给该 `<label>` 内的交互元素。因此，在输入框附近的操作被浏览器误认为是针对按钮的交互。

**修复方案**：
将包裹这部分组件的 `<label>` 替换为了纯粹的 `<div>` 容器，并使用 CSS Flexbox 重新对齐了样式。现在鼠标只有在确切 Hover 到按钮本身时，按钮才会产生交互反馈，彻底消除了“幽灵”动画。

## 2. 自动提取术语功能的设计逻辑

**疑问**：目前的提取功能是将整本 EPUB 的全文都发送给 AI 吗？

**答案：不是的。系统仅发送了原书的前部内容（截取前约 15000 个字符）。**

这一设计主要基于以下三个维度的考量：

1. **防止 Context Window（上下文窗口）溢出**：
   一整本 EPUB 电子书通常包含几十万字（对应数十万 Tokens）。如果将全量文本直接发送给普通的 LLM API（如 OpenAI 的标准模型或本地部署的模型），将直接超出其最大 Token 限制，导致 API 拒收并抛出错误。
2. **响应速度与 API 成本**：
   发送整本书进行深度名词提取的 Token 消耗极度昂贵，并且大模型处理超长文本的时间可能会长达数分钟。这极易导致前端请求发生超时（Timeout）断连。
3. **内容分布规律（首因效应）**：
   在绝大多数文学作品中，核心人物、关键地名、主要阵营与势力，都会在故事的前几个章节（即最初的一两万字）密集出场并建立世界观。截取前 15000 个字符已经足够高频地抓取到支撑全书翻译的 Glossary（核心术语表）。

**手动补充机制**：
如果遇到了出场极晚的重要角色（例如在第 30 章才首次出现的反派），目前的截断机制确实会漏掉该角色。为了解决这个问题，自动提取的结果并非直接“锁死”，而是会填充在前端的输入框中。用户可以随时**手动将漏掉的角色名补充到输入框里**，格式与自动提取的一致（例如：`LateHero=晚期英雄`），从而实现完美的术语统一。
